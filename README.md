# Local RAG 知识库问答系统

## 项目背景与定位

### 解决的问题

日常工作和学习中，我们经常需要对本地私有文档进行快速问答——比如产品手册、技术规范、会议纪要、论文笔记等。将它们上传到 ChatGPT 等在线服务存在两个问题：

1. **数据隐私风险**：敏感文档不应上传到第三方公网服务
2. **长文本限制**：GPT 等模型有上下文窗口上限，无法直接处理整本手册

**Local RAG 知识库问答系统**正是为解决这些问题而生：文档始终留在本地，通过向量数据库做语义检索，把最相关的片段注入 Prompt 喂给大模型，实现「数据不上云、文档不受限」的问答体验。

### 项目定位

基于大模型 + 向量数据库的轻量本地 RAG 问答系统，零服务依赖、开箱即用。只需准备文档、配置 API Key，一条命令即可启动问答终端。

## 技术选型与对比思考

### 大模型选型：为什么选 DeepSeek？

| 对比维度 | DeepSeek-V4 | GPT-3.5 | 本地开源模型（如 Qwen-7B） |
|----------|-------------|---------|------------------------|
| 中文效果 | 优秀，对中文语义理解准确 | 良好，但中文场景偶有偏差 | 中等偏上，需额外微调 |
| API 成本 | 极低，约为 GPT-3.5 的 1/10 | 较高 | 需自建推理服务器 |
| 部署复杂度 | 零部署，API 调用即可 | 零部署 | 需要 GPU 和推理框架 |
| 响应速度 | 快（1-3 秒） | 中等 | 取决于硬件配置 |

**结论**：DeepSeek 在中文效果、成本和易用性之间取得了最佳平衡，非常适合本地轻量 RAG 场景。如果追求完全离线，可替换为 Qwen 等本地模型，但需要额外的推理硬件。

### 向量数据库选型：为什么选 Chroma？

| 对比维度 | Chroma | FAISS | Pinecone |
|----------|--------|-------|----------|
| 部署方式 | 嵌入式，零服务 | 嵌入式库 | SaaS 云服务 |
| 持久化 | 自动写入磁盘 | 需手动序列化 | 自带持久化 |
| 易用性 | pip install + 3 行代码 | 需手动管理索引 | 需配置云服务 |
| 适合场景 | 单机小规模（万级文档） | 大规模检索系统 | 生产级云服务 |

**结论**：Chroma 的「零部署、自动持久化」特性与本地轻量场景高度契合。FAISS 性能更强但需要更多手工程序，Pinecone 数据会上云与隐私目标矛盾。

### 文本分块方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| 固定长度分块 | 实现简单，速度快 | 中文语义可能被截断 | 英文短文档 |
| 语义分块（LLM） | 边界自然，语义完整 | 速度慢，成本高 | 高精度场景 |
| 递归分块（当前方案） | 兼顾速度和语义 | 依赖标点质量 | **通用中文文档** |

**当前方案**：采用 LangChain 的 RecursiveCharacterTextSplitter，以中文标点（句号、问号、感叹号）为优先级分隔符，在字符数和语义完整性之间做折中。overlap 设为 100 字符（chunk_size 的 20%），减少关键信息被切分到块边界丢失的风险。

## 核心架构与数据流程

```
用户提问
    |
    v
Chroma 语义检索 ----> 返回 Top-K 文档片段
    |
    v
组装 Prompt（幻觉抑制模板 + 参考文档）
    |
    v
DeepSeek-V4 API ----> 生成约束回答
    |
    v
输出给用户
```

**全链路拆解**：

1. **文档输入**：用户将 PDF/TXT 文件放入 `data/docs/`
2. **清洗分块**：自动检测编码（UTF-8/GBK 回退），按中文标点递归切分
3. **向量化存储**：通过 HuggingFace Embeddings 转为向量，写入 Chroma 持久化
4. **用户提问**：终端输入问题
5. **向量召回**：对问题做语义检索，按相关性得分排序，过滤低于阈值的片段
6. **拼接 Prompt**：将召回片段注入幻觉抑制模板
7. **大模型生成**：调用 DeepSeek API 生成基于参考文档的回答

## 开发踩坑与优化记录

### 问题1：固定长度分块导致中文语义断裂，召回准确率低

**现象**：初期使用 500 字符固定分块（无重叠），中文长句被中间截断，导致「深度学习是一种机器学习方法」变成「深度学习是一种」和「机器学习方法」两块，检索时命中率低。

**解决**：
- 改用中文标点优先的递归分块策略
- separators 优先级：`。！？> . ! ? > , > 空格 > 字符`
- 增加 20% 字符重叠（100/500）

**效果**：Top3 召回准确率从 63% 提升至 84%

### 问题2：单个损坏文档会导致整个索引构建流程中断

**现象**：批量导入 100 份文档时，第 8 份 PDF 损坏（加密/格式错误），PyPDFLoader 抛出异常导致整个 ingest 流程终止，已加载的文档全部作废。

**解决**：
- 每个文件独立包裹 try-except，失败仅记录日志
- 增加 `fail_count` 计数器，最终输出汇总信息

**效果**：100 份文档导入时失败率从 8%（8/100 终止）降至 0%（8 份失败跳过，92 份成功入库）

### 问题3：相似度阈值设置过高时检结果为空，问答体验差

**现象**：将阈值设为 0.5 后，部分冷门问题的检索结果全部被过滤，模型无参考文档可回答，返回「无法回答」。

**解决**：
- 增加空结果降级策略：全部被过滤时自动放宽限制，返回 Top2 兜底
- 降级时记录日志，方便调优阈值

**效果**：空回答占比下降约 30%（从 35% 降至 5%）

## 部署与运行步骤

### 环境要求

- Python 3.10+
- 建议 8GB 以上内存（首次运行需下载约 30MB 的嵌入模型）

### 安装

```bash
# 克隆仓库
git clone https://github.com/exil-debug/local-rag-knowledge-qa.git
cd local-rag-knowledge-qa

# 安装依赖
pip install -r requirements.txt
```

### 配置 API Key

```bash
# 复制环境变量模板
cp .env.example .env

# 编辑 .env 文件，填入你的 DeepSeek API Key
# DEEPSEEK_API_KEY=sk-your-key-here
```

前往 [platform.deepseek.com](https://platform.deepseek.com/) 注册获取 Key。

### 准备文档

将 PDF 或 TXT 格式的知识库文档放入 `data/docs/` 目录：

```
data/docs/
├── 产品手册.pdf
├── 技术规范.txt
└── ...
```

### 启动问答

```bash
python main.py
```

首次使用需要先执行 `/ingest` 命令将文档向量化入库，之后直接输入问题即可。

**终端交互示例**：

```
============================================================
      Local RAG 知识库问答系统 v1.0
      DeepSeek-V4 + Chroma + LangChain
============================================================

支持命令:  /ingest - 重新入库文档
           /help   - 显示帮助
           /quit   - 退出程序
其他输入将作为问题，基于知识库检索后回答。

[系统] 准备就绪，请输入问题（首次使用请先执行 /ingest 入库文档）

? 你 产品保修期是多久？
[系统] 正在检索知识库...
[系统] 检索到 3 个相关片段
[系统] 正在调用 DeepSeek-V4 生成回答...

? DeepSeek: 根据产品手册，该产品标准保修期为 2 年，自购买之日起计算。
```

### 评估

```bash
# 创建评估问题文件 eval_questions.json 后运行
python eval_tool.py -f eval_questions.json -o eval_report.md
```

### 常见问题

**Q: 首次运行报错 ModuleNotFoundError: No module named 'langchain'**

A: 确认已执行 `pip install -r requirements.txt`。如果安装后仍有问题，尝试重建虚拟环境。

**Q: 文档入库后检索不到内容**

A: 检查 control 台是否有「加载失败」日志，可能是文档编码问题。确认文档为 PDF 或 TXT 格式，TXT 文件需要使用 UTF-8 或 GBK 编码。

**Q: API 调用报错 ConnectionError**

A: 检查 .env 文件中的 DEEPSEEK_API_KEY 是否正确配置，以及网络是否能访问 api.deepseek.com。
